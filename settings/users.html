auth

Predefined string which specifies the overall authentication mode of the site. Default: inherit

Value Description
inherit Default. If the site is nested, inherit the parent site's user behavior, otherwise none.
none Explicitly disabled user and/or session functionality.
required The entire site is private access only, requiring user authentication for all endpoints.

roles

Changing and refactoring role names is a safe operation, as long as you also update any code references to them. Role names are not associated with data.

Roles contain 1 or more permission rules. Permission rules define what is allowed for a role. Most commonly permission rules map to model data operations.

Scripts and templates code against role names. In other words: "Is role ABC active for the current user/session". The permission rules that a role contains consists of two parts: 1) Resource Rule, 2) Operation(s) Allowed 


roles:
  [ROLE_NAME]:
    [RESOURCE_RULE]: [OPERATIONS_ALLOWED]

Operation Names

The available permission operations are: all, create read, update, delete, state

Most operations map directly to an HTTP method. For predictable handling of endpoints that need to be role-constrained, we recommend following this general pattern.

Operation Description HTTP
all All operations are allowed.
create Create operations are allowed POST
read Read operations are allowed.
Depending on the case "read" may also refer to "get", "load", "accessible", or "visible". 		GET
update Update operations are allowed.
May also refer to "editing" or "modifying" something that already exists. 		PUT
delete Delete operations are allowed.
May also refer to quasi-deletes, such as an operation that could be rolled back. 		DELETE
state State change operations are allowed.
This is a special case of "update", where a significant "state change" is also taking place. A state change usually involves concepts regarding "status", "publish", or "lifecycle" -- typically auditing or event logging takes place as well. 		PATCH
list List operations are allowed.
List operations are "bulk" operations, and may refer to queries, or anything that returns or operates on potentially multiple results.

Naming Recommendations

We recommend using a singular form for role names. This will help to clearly distinguish role names from the plural form recommended for group names.

groups

Unlike role names, group names are not freely changeable, especially if they have been used and associated with existing users.

Groups contain 1 or more named roles. Users are assigned to groups. Scripts and templates are unaware of groups -- they are only aware of the roles activated by groups.

Remember, groups are associated with all of your user data. Freely change and refactor your role names (and the code that uses them, of course), but remember that changing in-use group names will alter access for users associated with those groups. In short: changing group names is not recommended.

Full Group Example

Beneath the name of group, list out the roles it contains. 


groups:
  administrators:
    - admin
  editors:
    - task-editor
Naming Recommendations

As groups represent a collection of many users, we recommend using the plural form of group names. This will also more clearly distinguish group names from the singular form recommended for role names.
